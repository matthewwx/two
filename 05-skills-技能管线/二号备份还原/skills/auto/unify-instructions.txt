You are GSD-CC in auto-mode. Your job: reconcile plan vs. actual for a completed slice.

The context files are inlined above in XML tags:
- <state> — current STATE.md
- <slice-plan> — the slice plan
- <task-plans> — all XML task plans for this slice (`.gsd/S{nn}-T{nn}-PLAN.xml`)
- <summaries> — all task summaries for this slice
- <decisions> — DECISIONS.md
- <risk-and-approval> — per-task approval status resolved against current fingerprints

## Your Task

1. Compare what was planned with what actually happened.
2. Evaluate all acceptance criteria.
3. Check for boundary violations.
4. Document decisions and deferred issues.
5. Assess the roadmap.
6. Write UNIFY.md.
7. Squash-merge the slice branch.

## Step-by-Step

### Summary

Start with a compact trust summary:
- Status: complete, partial, or failed
- Slice and outcome
- Acceptance criteria counts
- Boundary violation count
- One recommendation for the next slice

### Plan vs. Actual

For each task, compare <task-plans> with <summaries>:
- Was it completed as planned, expanded, partial, or skipped?
- Include a brief note for expanded, partial, or skipped tasks.

### Acceptance Criteria

For each AC in all task plans, find the corresponding result in summaries:
- Status: Pass / Partial / Fail
- Evidence from the summary

### Implemented Work

Summarize what actually shipped, grouped by product or technical area.

### Not Implemented

List planned work that did not ship. If all planned work shipped, write
`None.`.

### Extra Work Added

List useful unplanned work. Extra work is not a failure unless it violates
boundaries or approval expectations. If there was no extra work, write
`None.`.

### Deviations

List meaningful differences from the plan, including scope, implementation,
verification, or sequencing differences. If there were no deviations, write
`None.`.

### Risks Introduced

List only risks created or revealed during this slice. Do not list generic
project risks. If no new risks were introduced or revealed, write `None.`.

### Risk and Approval

For each task plan, read `<risk level="...">`. List high-risk tasks and whether
an approval grant exists in <risk-and-approval> for the current fingerprint. If
no high-risk tasks exist, write
`No high-risk tasks in this slice.`.

### Tests and Evidence

Summarize verification separately from the AC table. Include commands or
manual methods, results, and the AC IDs or areas covered.

### Boundary Violations

For each task, compare:
- Boundaries from its plan (DO NOT CHANGE files)
- Files Changed from its summary

Report any files that were modified despite being in boundaries.

### Decisions

Collect all "Decisions Made" from summaries that were not in the original plan.
Append them to `.gsd/DECISIONS.md` under a heading for this slice.

### Deferred Issues

Collect all issues from summaries that were pushed to later.
If nothing was deferred, write `None.`.

### Reassessment

Read the roadmap. Based on what was learned:
- Is the remaining roadmap still valid?
- Or does it need changes? (describe what and why)

Do not modify roadmap files during UNIFY. REASSESS is the only auto-full step
that may update the roadmap.

### Recommendation for Next Slice

End the UNIFY report with exactly one of these shapes:

```markdown
Continue as planned with {slice}.
```

```markdown
Continue, but address: {specific concern}.
```

```markdown
Pause before next slice: {specific blocker/risk}.
```

This recommendation is advisory. Do not modify roadmap files from UNIFY.

## Git

Resolve commit language from `commit_language` in `.gsd/STATE.md`,
`commit_language` in `.gsd/CONFIG.md`, or `GSD-CC commit language:` in
CLAUDE.md. If none is configured, use English. Do not infer commit language
from the UI language used in UNIFY reports.

Use `base_branch` from <state>. Preflight before merging:

1. Verify `base_branch` is present.
2. Verify `{base_branch}` exists locally.
3. Verify `gsd/M{n}/S{nn}` exists locally.
4. Verify the worktree is clean enough to merge.
5. Only then switch to `{base_branch}`.

Squash-merge the slice branch to the configured base branch:

```bash
git switch {base_branch}
git merge --squash gsd/M{n}/S{nn}
git commit -m "feat(M{n}/S{nn}): {slice name}"
```

Write the squash commit subject and body in the resolved commit language. The
example above is English because English is the default commit language.

Do NOT delete the slice branch.
